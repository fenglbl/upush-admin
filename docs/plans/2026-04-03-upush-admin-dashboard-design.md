## 1. 目标

构建 `upush-admin` 第一版后台首页，定位为：

- **以推送结果为主的数据看板**
- **日志为辅**
- **兼具少量工作台操作**

第一版的重点不是做完整后台，而是先做出一个：

1. 能直观看到最近推送效果
2. 能查看近几天推送趋势
3. 能快速发现异常
4. 能作为后续 admin 扩展的首页骨架

---

## 2. 第一版范围

### 2.1 包含内容

#### 首页（Dashboard）

1. **KPI 卡片区**
   - 今日推送总数
   - 成功数
   - 部分失败数
   - 失败数
   - 成功率

2. **趋势图区域**
   - 默认展示：近 7 天
   - 支持切换：
     - 近 7 天（`7d`）
     - 近 30 天（`30d`）
     - 近 24 小时 / 今日按小时（`24h`）
   - 折线数据：
     - 总推送数
     - 成功数
     - 失败数

3. **最近推送结果列表**
   - 标题
   - 时间
   - 状态（成功 / 部分失败 / 失败）
   - 简要结果信息

4. **最近异常日志摘要**
   - 最近若干条 `error / warn`
   - 展示时间、级别、摘要
   - 提供跳转到日志页入口

5. **快捷操作区**
   - 新建推送
   - 查看推送记录
   - 查看日志

---

### 2.2 第一版不做

以下内容明确不纳入第一版：

- 完整用户管理
- 完整设备管理
- 完整权限系统
- 完整日志检索页（可先只留入口）
- 复杂 BI 图表
- 多维筛选分析
- 首页拖拽布局
- 后端底层数据模型大改造

原则：**先跑起来，再扩展。**

---

## 3. 设计方向

### 3.1 首页定位

首页应该优先回答：

> 今天推送得怎么样？最近趋势如何？是否有明显异常？

因此信息优先级应为：

1. 推送结果 KPI
2. 推送趋势图
3. 最近推送结果
4. 异常日志摘要
5. 快捷操作

---

### 3.2 视觉风格

建议风格：

- 偏专业后台
- 卡片化布局
- 信息层级清晰
- 以数字与状态色表达信息
- 控制装饰感，不过度花哨

建议视觉关键词：

- **清晰**
- **稳定**
- **专业**
- **可读性强**

### 3.3 UI 决策（已确认）

#### 整体视觉方向

- 浅色后台为主
- 数据看板优先
- 推送结果为主、日志为辅
- 偏专业后台，不做大屏炫技风
- 以卡片、数字、状态色建立层级

#### 首页结构

建议首页采用三段式布局：

1. 顶部信息区（标题、环境、最后刷新时间、少量快捷操作）
2. 核心数据区（KPI 卡片 + 趋势图）
3. 明细区（最近推送结果 + 最近异常日志）

#### KPI 卡片风格

已确认采用：

- **整体简洁 + 成功/失败卡片带轻状态底色**

具体建议：

- 今日推送总数：白底卡 + 中性蓝灰点缀
- 成功数：极浅绿色底 + 深绿色数字
- 部分失败数：极浅橙色底 + 橙色数字
- 失败数：极浅红色底 + 红色数字
- 成功率：白底或极浅蓝底 + 蓝色数字强调

这种方式既能快速表达状态，又保持后台专业感。

#### 趋势图模块

- 图表标题：推送趋势
- 默认时间范围：近 7 天
- 支持切换：`7d / 30d / 24h`
- 图表类型：折线图
- 曲线：总推送数 / 成功数 / 失败数
- 图表卡片高度建议：320~380px

#### 明细区模块

左栏：最近推送结果
- 标题、时间、状态、摘要
- 采用简洁列表卡片，而不是首页表格化重布局

右栏：最近异常日志
- 仅展示 error / warn 摘要
- 保持辅助地位
- 提供“查看全部日志”入口跳转到独立日志页

#### 快捷操作区

首页仅保留少量工作台操作：

- 新建推送
- 查看推送记录
- 查看日志

避免首页退化为按钮墙。


### 4.1 首页布局

- 当前环境标签（dev / prod）

#### 第一屏：KPI 卡片

4. 失败数

#### 第三屏：双栏内容区

左侧：最近推送结果

## 6. 数据来源建议

### 6.1 首页主数据

首页主数据来源应优先围绕：

- 推送结果记录

用于生成：

- KPI 卡片
- 趋势图
- 最近推送结果

### 6.2 首页辅助数据

辅助数据来源：

- 日志系统

用于生成：

- 最近异常日志摘要

### 6.3 第一版原则

- 能聚合出来就先聚合
- 不为第一版首页大改历史数据结构
- 若部分统计暂时缺少精确来源，可先做保守聚合

## 8. 迭代顺序建议

### 阶段 1：后端接口

1. 明确推送结果可聚合的数据源
2. 明确日志摘要的读取方式
3. 实现 `/admin/dashboard`
4. 返回固定 shape

### 阶段 2：前端首页骨架

1. 基础布局
2. Header
3. KPI 卡片
4. 图表区域
5. 双栏列表区域
6. 快捷操作区

### 阶段 3：联调

1. 首页联调 dashboard 接口
2. 切换 `7d / 30d / 24h`
3. 校验空状态 / 错误状态 / 加载状态

### 阶段 4：视觉收尾

1. 卡片层级
2. 状态色统一
3. 图表配色统一
4. 间距与对齐优化

## 9. 风险与注意事项

### 9.1 风险

- 推送结果记录如果缺少统一聚合字段，统计需要先做兼容处理
- 近 30 天与 24h 维度切换需要统一数据 shape
- 首页不应直接依赖复杂日志全文检索逻辑

### 9.2 注意事项

- 第一版优先保证数据结构稳定
- 先保证首页能准确展示关键结果
- 日志详情能力留给独立页面
- 不在第一版引入过重权限体系

---

## 10. 结论

`upush-admin` 第一版将采用：

- **推送结果优先的数据看板首页**
- **默认近 7 天趋势，可切换 30 天与 24 小时**
- **日志仅做摘要辅助**
- **新增专用 dashboard 接口**
- **基于现有数据结构快速聚合实现**

这是一个偏稳健、可快速落地、可持续扩展的第一版方案。
